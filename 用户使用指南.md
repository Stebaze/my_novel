# 用户使用指南：ask-yiyi 视角的操作手册

> **定位**：面向作者的完整操作手册。以 **ask-yiyi 统一入口** 为视角组织——你只需要知道"说什么 → 发生什么"。具体 Skill 的内部实现见各自 `SKILL.md`；Agent 强制规则见 [`CLAUDE.md`](CLAUDE.md)；方法论见 [`framework/guides/`](framework/guides/)。

---

## 核心理念

**AI 不写正文。** 这个项目的角色是：

- **写之前**：帮你理清方向，产出"写作简报"——告诉你可以怎么写、用什么技法、注意什么
- **写之后**：帮你发现问题，产出"修改报告"——指出哪里卡、哪里崩、哪里角色行为不一致

**笔始终在你手里。** 简报不是必须照做的剧本，是可以偏离的起点。评审报告不是批评，是帮你看自己盲区的工具。

**ask-yiyi 统一入口**——会话启动后你不需要记任何命令，ask-yiyi 会扫描项目状态、给出 5 字段摘要、推荐下一步。你说"做什么"，ask-yiyi 帮你"找对工具"。

---

## ask-yiyi 是什么

**会话级统一入口**——启动后自动接管，完成以下事情：

1. **格式检测**（`migration-keeper detect-format`）—— 检查 `novel/` 目录是否使用最新格式
2. **兼容性检查**（`migration-keeper check-compat`）—— 旧格式给出迁移建议
3. **状态加载**—— 扫描 `_drafts/` 选最新草稿、读 `notes.md` / `session-context.md`、扫工件（`_briefs/`、`chapters/`、`_reviews/`）
4. **5 字段摘要**—— 输出当前进度（章节数 / 当前阶段 / 草稿目录 / 工作流状态 / 格式警告）
5. **智能推荐**—— 根据状态机直接给"← 💡 推荐"的下一步

**5 种操作模式**——见下节。

---

## 5 种操作模式

| 操作 | 触发词 | 用途 |
|------|--------|------|
| `init` | （会话启动时自动）| 加载项目状态、给出 5 字段摘要、推荐下一步 |
| `route` | `/ask-yiyi` / "创作工坊" / "工坊" / "现在做什么" | 列出主菜单并路由到具体 Skill |
| `next` | "继续" / "下一步" | 根据状态机直接给推荐操作（跳过菜单） |
| `qa` | `/ask-yiyi qa` / "怎么用" / "为什么" / "我该用哪个" / "help" | 答疑（只读答工具用法） |
| `qa check` | `/ask-yiyi qa check` / "检查" / "审计" | 异常审计（检查工件完整性 + 步骤跳跃） |

**默认行为**：会话启动后 `init` 自动跑一次，给你摘要和推荐。如果你只是想看菜单或答疑，说对应触发词。

## 三组主菜单

ask-yiyi 的主菜单按使用场景分组（**强依赖工作流的工具不上主菜单**，由上游自动调用）：

```
[继续写这本书]  (NO_PROJECT 时整组隐藏)
  1. plan-chapter           写第X章
  2. generate-chapter       AI 生成整章
  3. chapter-review         评审第X章
  4. publish-chapter        发布第X章
  5. import-chapter         导入已有章节

[创建新书]
  6. bootstrap-project      从已有原稿冷启动
  7. adaptation-workflow    改编已有作品

[其他工具]
  8. idea-explorer          发散性头脑风暴
  9. qing-novelist          12 维启发交谈 / 7 维作者分析
  10. (本组 qa)             答疑
  11. (本组 qa check)       异常审计

[元]
  99. 完整菜单              含内部组件（12-15）
  0. 退出
```

**完整菜单（编号 99）** 新增内部组件 12-15（`settings-manager` / `pre-flight-check` / `file-manager` / `migration-keeper`）——这些"强依赖工作流 · 通常由上游自动调用"，主菜单不展示。

**每个选项的推荐标记**：状态机命中的推荐项会被标为 `← 💡 推荐`。例如 `NO_HANDBOFF` 时选项 1 会带这个标记。

## 状态机 + 智能建议

ask-yiyi 在 init 时根据磁盘工件自动判断状态：

| 状态 | 触发条件 | 推荐 |
|------|---------|------|
| `NO_PROJECT` | 无 `novel/project-config.md` | bootstrap / adaptation（选项 6 / 7） |
| `NO_HANDBOFF` | 当前章节无 `chapter-{N}-handoff.md` | plan-chapter（选项 1） |
| `BRIEF_READY` | handoff 有，brief 有，但章节未写 | generate-chapter（选项 2） |
| `CHAPTER_WRITTEN_NO_REVIEW` | 章节已写，无评审 | chapter-review（选项 3） |
| `REVIEWED` | 评审已完成 | publish-chapter（选项 4） |
| `PUBLISHED` | 章节已发布 | 继续 plan-chapter（选项 1，写下一章） |
| `ANOMALY` | 异常状态（review 无 chapter / brief 缺 direction / chapter 缺 brief）| qa check（选项 11） |

每个状态只在菜单中标一次 `← 💡 推荐`，避免重复。

## 4 session 工作流在 ask-yiyi 视角下的串联

4-Skill 对称架构把"写一章"切到 4 个 session，每段独立 context：

```
SESSION 1 — 规划（plan）              SESSION 2 — 生成（generate）
  /plan-chapter {N}                      /generate-chapter {N}
  → 方向卡 + handoff                     → 简报 + 章节 + 评审 + Fix ≤ 2 轮

[可选] SESSION 3 — 人工复审           SESSION 4 — 发布
  /chapter-review {N}（writing）         /publish-chapter {N}
  → 评审 + 修订循环 + 定稿               → 设定合并 + 琉璃校验
```

**ask-yiyi 在哪里出现**：SESSION 1 / 2 / 3 / 4 启动时，ask-yiyi 都会先跑 init 给你 5 字段摘要和推荐。SESSION 1 的"规划"前 ask-yiyi 会确认 `NO_HANDBOFF` 才推荐 plan-chapter。SESSION 2 的"生成"前 ask-yiyi 会确认 `BRIEF_READY` 才推荐 generate-chapter。

**为什么切 session**：避免单一长 session 上下文膨胀导致规划/简报/正文/评审互相干扰。每段都是干净的小上下文，专注做好一件事。

**详细 walkthrough** 见各自 `SKILL.md`：
- `plan-chapter` → `.claude/skills/plan-chapter/SKILL.md`
- `generate-chapter` → `.claude/skills/generate-chapter/SKILL.md`
- `chapter-review` → `.claude/skills/chapter-review/SKILL.md`
- `publish-chapter` → `.claude/skills/publish-chapter/SKILL.md`

## QA 模式使用边界

ask-yiyi 的 `qa` 和 `qa check` 边界清晰：

| 模式 | 答什么 | 不答什么 |
|------|-------|---------|
| `qa`（"怎么用"）| 工具用法、触发词、菜单结构、工作流顺序 | bug 报告、需求收集 |
| `qa check`（"检查"）| 工件完整性、步骤跳跃、异常状态 | 写作内容质量评审 |

**❌ 不用 ask-yiyi 提 bug 单**——bug 报告走项目根目录的全局 `qa` Skill（与 ask-yiyi 的 qa 是同名但不同层），会进 GitHub Issues 流程。

**❌ 不用 ask-yiyi 做章节评审**——那是 `chapter-review` Skill 的事（`writing` / `ai-content` / `adaptation` 三 mode）。

## 草稿三层体系（横切概念）

不管用哪个 Skill 写，稿件都在三层体系中流转：

| 层 | 目录 | 性质 | 修改权限 |
|----|------|------|---------|
| **原稿** | `reference/manuscripts/` | 用户上传原始素材 | 🚫 禁止修改 |
| **草稿** | `novel/_drafts/{日期}/` | 多会话实验空间，独立设定副本 | ✅ 自由修改 |
| **正式稿** | `novel/chapters/` | 已发布版本 | ⚠️ 谨慎修改（锁定后不可回退） |

**迁移路径**：
- 原稿 → 草稿：`import-chapter`（单/少量章节）/ `bootstrap-project`（批量冷启动）
- 草稿 → 正式稿：`publish-chapter`（设定合并 + 草稿同步 + 琉璃校验）

**草稿管理**：初始化（`file-manager` `ensure-draft`）/ 写作期（独立设定副本互不干扰）/ 接受合并（`settings-manager` `merge`）/ 废弃（保留作为历史）

---

## 设定时间线管理（横切概念）

所有设定以**章节为版本标记**，每条信息标注"引入章节"。

- **读取**：评审/修改第 N 章时，只参考引入章节 ≤ N 的设定条目
- **修改**：修改设定时向前扫描——检查章节号 > 旧引入章节的已有章节是否因此冲突
- **冲突分级**：
  - 🔴 必须修（直接矛盾/能力越界）
  - 🟡 应修（关系跳跃/细节不一致）
  - 🟢 提示（风格微调/可选）
- **角色状态追踪**：每章结束时记录出场角色的 9 维状态（时间/空间/身体/心理/关系/目标/情绪/持续行动/关键物品）

详细操作（读取合并优先级、向前扫描执行步骤、差异对比）见 `settings-manager` Skill。

---

## 降级策略与阻断等级（横切概念）

所有 Skill 在发现所需文件缺失时，遵循统一顺序：

1. **先补齐**：调 `file-manager` 尝试从上层补齐
2. **补齐失败才降级**：明确标注"⚠️ 文件 X 缺失：能力 Z 已降级为 Y"
3. **降级不静默**：每次降级在阶段输出中体现

**阻断等级**：

| 等级 | 含义 | 例子 |
|------|------|------|
| 🚫 硬阻断 | 不能继续，必须解决 | author-voice.md 缺失 → 禁止进入简报生成 |
| 🟡 软阻断 | 警告 + 用户确认后放行 | 出场角色档案大面积缺失 |
| ⚠️ 提醒 | 标注影响范围，不阻断 | 参考指南缺失 |

**强制规则**：Skill 调用失败必须 🚫 硬阻断，禁止 ⚠️ 降级后继续。仅文件/Guide 缺失可选 ⚠️。8 条强制规则字面见 [`CLAUDE.md`](CLAUDE.md)。

## 常见问题 FAQ

### 我该选哪条路径开始项目？

| 你的情况 | 选择 |
|---------|------|
| 已有成稿（哪怕只有几章）| 冷启动（"初始化项目"）— 系统从你的正文反推大纲和设定 |
| 完全从零开始 | 新书项目（"我要写一本小说"）|
| 有原作要改写 | 改编（"改编这个作品"）|

### 冷启动（bootstrap）和导入章节（import）有什么区别？

**冷启动** = 完整项目初始化——批量导入 → 全文逆向分析 → 作者确认 → 工件生成。适合"第一次把稿件纳入系统"。

**导入章节** = 轻量操作——导入单个/少量章节到已有项目的草稿，自动扫描设定，注册评审队列。适合"项目已在运行，只想补几章"。

### 我可以跳过规划直接写吗？

不可以。写前规划是强制管道——前置检查确保设定就绪，五更交谈确保方向清晰，简报确保你有足够的信息支撑写作。

### 我可以跳过评审直接发布吗？

不可以。评审确保设定一致性和文本质量——跳过评审可能导致后续章节基于有问题的设定继续写，越错越远。

### 简报中的参考示例我必须照着写吗？

不。参考示例是"如果你不确定怎么写，可以这样写"的参考，不是模板。你完全可以偏离它，用自己的方式写。偏离的部分会在评审中标注，但不代表错误。

### 评审说的问题我必须全改吗？

不。🔴 强烈建议修，🟡 你可以选择修或不修，⚠️ 完全由你决定。使用「这个问题不改了」可以标记为"作者确认不修改"，不阻塞定稿。

### 我可以只评审部分内容吗？

可以。说「只查场景三」即可指定评审范围。

### AI 生成章节和人工写章节的评审有什么区别？

`chapter-review` 提供 3 mode：
- `writing`（默认）— 人工写章节的标准评审
- `ai-content` — AI 生成章节，含场景级机器化校验（事件落地/场景连贯/突兀收束/POV 连续/伏笔核对）
- `adaptation` — 改编章节，含原作对齐检查（严格/平衡/宽松）

### ask-yiyi 的"qa"和项目根目录的"qa" 是什么关系？

ask-yiyi 的 `qa` 是**会话内答疑**——只读答工具用法，不提 bug 单。  
项目根目录的全局 `qa` 是 **GitHub Issue 收集器**——把 bug / 需求对话式收集成 GitHub Issue。  
两者同名但不同层：ask-yiyi 的 qa 永远在 Claude 会话内；全局 qa 会落盘到 GitHub。
